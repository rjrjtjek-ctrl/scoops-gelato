## 🚨 절대 하지 마라 (피로 배운 것들)

### 1. `iframe.contentWindow.print()` 절대 금지
**무슨 일이 있었나**: 영수증 인쇄를 iframe으로 바꿨더니 관리자 대시보드가 15초 이상 멈췄다. 폴링 멈추고, UI 얼고, 인쇄 대화상자 닫을 때까지 아무것도 안 됨.

**왜 그런가**: `iframe.contentWindow.print()`는 **메인 페이지의 JS 스레드를 블로킹**한다. popup의 `print()`는 팝업 스레드만 블로킹하므로 메인 페이지는 정상 작동.

**정답**: `window.open()` 팝업에서 `print()` 호출. 무조건 이 방식.

---

### 2. `@page { size: 72mm 200mm }` — 높이 고정 금지
**무슨 일이 있었나**: 열전사 프린터가 200mm 분량의 종이를 출력했다. 영수증 내용은 80mm인데 나머지 120mm는 빈 종이. CEO가 "절취선 문제"로 보고.

**왜 그런가**: 열전사 프린터는 CSS @page 높이값 그대로 종이를 밀어낸다. 내용이 짧아도 지정된 높이까지 빈 공간을 채움.

**정답**: `@page { margin: 0; padding: 0; }` 만 사용. 높이 지정 없음. 프린터가 내용 끝에서 자동 절단.

---

### 3. 팝업 2개로 영수증 2장 인쇄 — 비효율
**무슨 일이 있었나**: 주방용/손님용을 팝업 2개로 순차 인쇄했더니 프린트 다이얼로그가 2번 나옴. 각 팝업 열기/닫기 오버헤드 + 대화상자 2번 = 느리고 번거로움.

**정답**: CSS `page-break-after: always`로 하나의 팝업에 2장을 넣으면 print() 1번으로 프린터가 자동 절단하여 2장 출력. 현재 이 방식 사용 중.

**구조**:
```
buildReceiptPage(order, "주방용") → HTML 조각
buildReceiptPage(order, "손님용") → HTML 조각
buildCombinedReceiptHtml(order) → 위 두 조각을 page-break로 분리한 완성 HTML
printViaPopup(html) → 팝업에서 print()
```

---

### 4. 성능 최적화하다 기존 기능 깨뜨리기 — CEO 극혐
**무슨 일이 있었나**: 인쇄 속도를 빠르게 만든다고 popup→iframe으로 바꿨더니 15초 멈춤. 2장 출력을 1장 합본으로 바꿨더니 빈 종이 나옴. CEO가 "뭐 하나 고치면 뭐 하나 고장나고"라고 강하게 지적.

**교훈**:
1. 하나 고치기 전에 **현재 코드를 반드시 먼저 읽어라**
2. 고친 후 **전체 플로우를 반드시 테스트**해라 (QR스캔→메뉴→카트→주문→관리자감지→인쇄)
3. "더 나은 방법"이 떠올라도 **현재 작동하는 것을 먼저 존중**해라
4. 변경 범위를 최소화해라. 한 번에 여러 군데 고치지 마라

## ⚡ 성능 — 이미 싸운 전쟁

### 30초 → 7초 (서버)
| 전 | 후 | 방법 |
|----|----|----|
| Supabase INSERT 3번 | RPC 1번 | `create_order_with_items` RPC 함수 |
| Supabase SELECT 2번 | RPC 1번 | `get_orders_with_items` RPC 함수 |
| Vercel 미국 동부 (iad1) | 서울 (icn1) | `vercel.json` `"regions": ["icn1"]` |

### 7초 → ~2초 (프론트엔드)
| 전 | 후 | 방법 |
|----|----|----|
| API 응답 기다림 → 페이지 이동 | 즉시 영수증 표시 | Optimistic UI (6ms) |
| /complete 페이지로 redirect | 같은 컴포넌트에서 상태 변경 | `orderComplete` state |
| 관리자 3초 폴링 | 700ms 폴링 | `setInterval(fetchOrders, 700)` |

### 남은 병목
- **Supabase DB가 Sydney** — 서울 Vercel에서 왕복 ~200ms 추가. 서울 리전 이전하면 각 API 100-150ms 빨라짐
- **폴링 방식의 한계** — 평균 350ms 지연. SSE/Realtime으로 바꾸면 즉시 감지 가능
- **프린터 물리적 출력** — 이건 코드로 해결 불가

## 🎯 이 프로젝트의 바이브

### CEO(정석주 대표)는 이런 사람이다
- "긴급"이라고 하면 **진짜 긴급**이다. 지금 매장에서 손님이 기다리는 중일 수 있다.
- 동작하는 결과를 빨리 보고 싶어한다. 설명보다 실행.
- "수익을 보장합니다" 같은 과장 문구 **절대 금지**. 가맹사업법도 위반이고 CEO 성향에도 안 맞음.
- 한 번 돌아가던 게 깨지면 **매우** 불쾌해한다. 테스트 안 하고 배포하면 신뢰 잃음.
- 정보공개서가 미등록 상태라 "정보공개서 확인하세요" 같은 콘텐츠 만들면 안 됨.

### 코드 수정의 대원칙
1. **읽고 → 이해하고 → 수정해라.** 파일 안 읽고 수정하면 100% 사고남.
2. **범위 최소화.** 3개 파일 동시에 고치지 말고 1개씩.
3. **배포 전 빌드.** `npx next build` 에러 없는지 확인.
4. **배포 후 테스트.** 실제 사이트에서 주문 한 번 해봐라.
5. **CLAUDE.md 업데이트.** 결정사항 바뀌면 반영해라. 이게 다음 세션의 기억이다.

### UX 결정 — 건드리지 마라
| 결정 | 이유 | 변경 가능? |
|------|------|-----------|
| 토글/아코디언 메뉴판 | 주류 15종이라 리스트로 나열하면 스크롤 죽음 | ❌ CEO 확정 |
| 주류 구경 자유 + 담기에서만 연령확인 | 메뉴 보는 건 자유, 법적으로도 구매 시점 확인이면 충분 | ❌ CEO 확정 |

## 🔧 인쇄 시스템 — 지뢰밭 안내

이 프로젝트에서 **가장 많은 삽질**이 일어난 곳이 영수증 인쇄다.

### 현재 정답 (건드리지 마라)
```
주문 감지 → printReceipt(order) → 큐에 1건 추가
→ processNextPrint() → buildCombinedReceiptHtml(order)
→ printViaPopup(html) → 팝업에서 print() 1번
→ 프린터가 page-break 기준으로 2장 자동 절단
```

### 안전장치 7개 — 다 이유가 있다
| 번호 | 뭘 막는가 | 빼면 어떻게 되나 |
|------|----------|----------------|
| FIX1 isFirstLoad | 페이지 새로고침 시 기존 주문 전부 재인쇄 | 30장 연속 출력 |
| FIX2 팝업 차단 감지 | Chrome 팝업 차단 시 무한 대기 | 인쇄 멈추고 에러 없음 |
| FIX3 인쇄 큐 (mutex) | 동시 print() 호출 | 프린터 먹통 |
| FIX4 네트워크 복구 제한 | 연결 끊겼다 복구 시 밀린 주문 전부 인쇄 | 종이 100장 나옴 |
| FIX5 절전모드 감지 | POS가 절전 후 깨어날 때 폴링 안 됨 | 주문 놓침 |
| FIX6 print() 중복 방지 | onload + setTimeout 둘 다 실행 | 같은 영수증 2번 인쇄 |
| FIX7 수동 재인쇄 | 인쇄 누락/장애 시 재출력 수단 없음 | CEO가 수동으로 기록 |

### POS 환경 주의
- **Windows 7 + Chrome 109** — 최신 JS 문법 일부 안 됨
- 기존 LOVE CHECK 프린터는 COM1 독점 → 웹에서 접근 불가
- CPP-3000(USB)만 웹 인쇄 가능
- `--kiosk-printing` 플래그로 프린트 다이얼로그 생략
